### 2.4 核心技术实现

#### 预算单号生成

```java
// 格式：BG-YYYYMM-XXX，如 BG-202606-001
String sheetNo = String.format("BG-%s-%03d", 
    DateFormatUtils.format(new Date(), "yyyyMM"), sequenceNumber);
```

#### 状态机流转（双字段设计）

```
status: Draft → Completed → Pending_Review（approval_stage: Dept → Branch → HQ → Approved）
任意审批级 → Rejected（驳回，回退到编制人员）
```

- 完成编制：校验明细完整性，Draft/Rejected → Completed
- 提交审核：校验状态为 Completed → 更新为 Pending_Review + approval_stage=Dept，启动 Flowable 流程
- 部门经理通过：approval_stage 从 Dept → Branch，完成当前流程任务
- 分公司本部通过：approval_stage 从 Branch → HQ，完成当前流程任务
- 总公司本部通过：status → Approved，approval_stage → None，预算锁定
- 任一级别驳回：校验驳回理由必填（5-500字符）→ status → Rejected，approval_stage → None，回退流程到编制人
- 编辑驳回记录：status 自动重置为 Draft，需重新完成编制

#### 差异自动计算

```java
// 差异金额 = 预算金额 - 实际金额
varianceAmount = budgetAmount - actualAmount;
// 差异率 = (差异金额 / 预算金额) × 100%
varianceRate = budgetAmount != 0 ? (varianceAmount / budgetAmount * 100) : 0;
```

#### 批量保存策略

1. 先删除该预算单的所有旧明细
2. 批量插入新明细
3. 事务控制（@Transactional）

## 五、Flowable 工作流集成

### 5.1 三级审批流程定义

```
开始事件 → 部门经理审批（用户任务） → 分公司本部审批（用户任务） → 总公司本部审批（用户任务） → 结束事件
```

每级审批均配置通过/驳回排他网关：
- 通过：流转到下一级审批
- 驳回：回退到编制人员（部门预算编制节点）

### 5.2 节点配置

| 节点 | ID | 候选组 | 表单Key |
|------|------|--------|--------|
| 部门经理审批 | deptLeaderReview | ROLE3（部门经理） | budget_approval_form |
| 分公司本部审批 | branchLeaderReview | ROLE4（分公司本部） | budget_approval_form |
| 总公司本部审批 | hqLeaderReview | ROLE5（总公司本部） | budget_approval_form |

### 5.3 角色配置

| 角色 | role_id | role_key | 候选组 | 审批级别 |
|------|---------|----------|--------|--------|
| 部门经理 | 3 | dept_leader | ROLE3 | 第1级 |
| 分公司本部 | 4 | branch_leader | ROLE4 | 第2级 |
| 总公司本部 | 5 | hq_leader | ROLE5 | 第3级 |

### 5.4 流转网关

| 条件 | 流向 |
|------|------|
| result=='pass' | 下一级审批（或结束事件） |
| result=='reject' | 退回编制人（deptBudgetPrepare） |

### 5.5 流程启动代码

```java
// 提交审核时启动流程，传入候选组变量
Map<String, Object> variables = new HashMap<>();
variables.put("deptLeaderGroup", "ROLE3");
variables.put("branchLeaderGroup", "ROLE4");
variables.put("hqLeaderGroup", "ROLE5");

ProcessInstance processInstance = runtimeService.startProcessInstanceByKey(
    "budgetProcess", sheetNo, variables);
budgetPreparation.setProcessInstanceId(processInstance.getId());
```

### 5.6 审批通过/驳回

```java
// 审批通过：完成当前任务，设置 result=pass，流程自动流转到下一级
Task task = taskService.createTaskQuery()
    .processInstanceId(processInstanceId)
    .singleResult();
Map<String, Object> vars = Map.of("result", "pass");
taskService.complete(task.getId(), vars);

// 驳回：设置 result=reject，流程回退到编制人
Task task = taskService.createTaskQuery()
    .processInstanceId(processInstanceId)
    .singleResult();
Map<String, Object> vars = Map.of("result", "reject");
taskService.complete(task.getId(), vars);
```

## 六、权限控制方案

### 6.1 功能权限

编制操作使用 `@SaCheckPermission` 注解控制 API 访问：

```java
@SaCheckPermission("system:preparation:submit")
@PostMapping("/submit/{id}")
public R<Void> submit(@PathVariable Long id) { ... }

```

### 6.2 数据权限

使用 `@DataPermission` + `@DataColumn` 注解自动注入过滤条件：

```java
@DataPermission({
    @DataColumn(key = "deptName", value = "org_id")
})
Page<BudgetPreparationVo> selectPageList(...);
```

### 6.3 数据范围配置

| 代码 | 名称 | SQL条件 |
|------|------|---------|
| 1 | 全部数据权限 | 无限制 |
| 2 | 自定义数据权限 | org_id IN (自定义列表) |

| 5 | 仅本人 | create_by = 当前用户 |

## 七、技术特性

### 7.1 事务控制

所有写操作使用 `@Transactional` 注解，确保数据一致性。

### 7.2 防重复提交

关键接口使用 `@RepeatSubmit` 注解。

### 7.3 参数校验

- BO 使用 JSR-303 注解：@NotNull, @NotBlank, @Min, @Max, @DecimalMin
- 分组校验：AddGroup（新增）、EditGroup（修改）

### 7.4 日志记录

使用 `@Log` 注解记录操作日志，包含操作标题和业务类型。

### 7.5 异常处理

使用 `ServiceException` 抛出业务异常，统一异常提示。
