|------|---------|------|
| admin / hq_manager | 所有部门 | 总公司管理员/总部管理者可查看全公司数据 |
| branch_manager | 分公司下所有部门 | 分公司管理者只能查看本部门及子部门数据 |
| 其他角色 | 仅自己部门 | 部门领导只能查看本部门数据 |

**实现逻辑**:
```javascript
// 在 created 钩子中调用 getUserInfo()
getUserInfo() {
  this.userRoles = this.$store.state.user.roles || [];
  this.userDeptId = this.$store.state.user.deptId;
  
  if (this.userRoles.includes('admin') || this.userRoles.includes('hq_manager')) {
    // 总公司：加载所有部门树
    this.loadDeptTree();
  } else if (this.userRoles.includes('branch_manager')) {
    // 分公司：加载分公司下所有部门
    this.loadDeptTree(this.userDeptId);
  } else {
    // 部门领导：固定为自己部门，不可修改
    this.queryParams.orgId = this.userDeptId;
  }
}
```

## 二、后端需要开发的接口

### 1. 批量审批接口
```java
// POST /system/preparation/batchApprove
@RequiresPermissions("system:preparation:approve")
@PostMapping("/batchApprove")
public AjaxResult batchApprove(@RequestBody BatchApproveBo bo)

// BatchApproveBo 结构
{
  List<Long> ids;  // 预算单ID列表
  String remark;   // 审批意见（可选）
}
```

### 2. 审批通过接口
```java
// POST /system/preparation/approve/{id}
@RequiresPermissions("system:preparation:approve")
@PostMapping("/approve/{id}")
public AjaxResult approve(@PathVariable Long id, @RequestParam(required = false) String remark)
```

**业务逻辑**:
1. 校验预算单状态是否为 `Pending_Review`
2. 更新状态为 `Approved`
3. 记录审批时间和审批人
4. 如果有关联的Flowable流程，完成任务
5. 发送通知给编制人

### 3. 审批驳回接口
```java
// POST /system/preparation/reject/{id}
@RequiresPermissions("system:preparation:approve")
@PostMapping("/reject/{id}")
public AjaxResult reject(@PathVariable Long id, @RequestParam String reason)
```

**业务逻辑**:
1. 校验驳回理由不能为空
2. 更新状态为 `Rejected`
3. 更新 `rejectReason` 字段
4. 设置 `rejectLevel`（根据审批人角色判断是HQ还是Branch）
5. 计算截止时间（根据驳回层级自动计算）
6. 设置当前处理人为编制人
7. 如果有关联的Flowable流程，完成任务并回退
8. 发送邮件/消息通知编制人

### 4. 获取校验结果接口
```java
// GET /system/preparation/validation/{sheetId}
@RequiresPermissions("system:preparation:query")
@GetMapping("/validation/{sheetId}")
public AjaxResult getValidationResults(@PathVariable Long sheetId)
```

**返回数据结构**:
```json
{
  "code": 200,
  "data": [
    {
      "ruleCode": "VAL_001",
      "ruleName": "预算超标检查",
      "riskLevel": "HIGH",
      "subjectCode": "3001001",
      "subjectName": "广告费",
      "budgetAmount": 50000.00,
      "errorMessage": "预算金额超出合理范围，请核实"
    }
  ]
}
```

**校验逻辑**:
1. 读取 `budget_validation_rule` 表中的启用规则
2. 根据规则的 `validation_scope` 过滤适用的科目类型
3. 执行校验公式或阈值比较
4. 返回所有未通过的校验结果

### 2. 角色数据范围配置
系统支持5种数据范围：

| 代码 | 名称 | SQL条件 |
|-----|------|--------|
| 1 | 全部数据权限 | 无限制 |
| 2 | 自定义数据权限 | `org_id IN (自定义部门列表)` |
| 3 | 本部门数据权限 | `org_id = 当前用户部门ID` |
| 4 | 本部门及以下数据权限 | `org_id IN (当前部门及子部门)` |
| 5 | 仅本人数据权限 | `create_by = 当前用户ID` |

## 四、校验规则引擎设计

### 1. 校验规则表结构
已在 `budget_validation_rule` 表中定义：

```sql
CREATE TABLE `budget_validation_rule` (
  `id` BIGINT(20),
  `rule_code` VARCHAR(50),          -- 规则编码
  `rule_name` VARCHAR(100),         -- 规则名称
  `rule_category` VARCHAR(20),      -- 规则类别
  `threshold_value` DECIMAL(10,2),  -- 阈值
  `risk_level` VARCHAR(10),         -- 风险等级
  `validation_scope` VARCHAR(20),   -- 校验范围
  `is_strict` TINYINT(1),           -- 是否严格校验
  `rule_formula` VARCHAR(500),      -- 校验公式
  `rule_description` VARCHAR(500),  -- 规则描述
  `error_message` VARCHAR(500),     -- 错误提示
  `is_active` TINYINT(1)            -- 是否启用
);
```

### 2. 校验规则示例

| 规则编码 | 规则名称 | 类别 | 阈值 | 范围 | 严格 | 说明 |
|---------|---------|-----|------|------|------|------|
| VAL_001 | 预算超标检查 | BUDGET_OVER | 20% | ALL | 否 | 偏差超过±20% |
| VAL_002 | 汇总逻辑检查 | LOGIC_ERROR | - | ALL | 是 | 明细合计=总额 |
| VAL_003 | 科目有效性检查 | SUBJECT_ERROR | - | ALL | 是 | 科目必须存在 |
| VAL_004 | 必填字段完整性 | MISSING_DATA | - | ALL | 是 | 必填字段非空 |
| VAL_005 | 历史数据对比 | HISTORICAL_COMPARE | 30% | EXPENSE | 否 | 同期波动±30% |
| VAL_006 | 收入类公式校验 | FORMULA_CHECK | - | INCOME | 否 | 收入>=0 |
| VAL_007 | 费用合理性检查 | BUDGET_OVER | 50% | EXPENSE | 否 | 费用增长<50% |

### 3. 校验执行流程

```java
public List<ValidationResult> validateBudget(Long sheetId) {
    List<ValidationResult> results = new ArrayList<>();
    
    // 1. 获取启用的校验规则
    List<BudgetValidationRule> rules = validationRuleMapper.selectList(
        new LambdaQueryWrapper<BudgetValidationRule>()
            .eq(BudgetValidationRule::getIsActive, 1)
    );
    
    // 2. 获取预算明细数据
    List<BudgetPreparationDetail> details = detailMapper.selectList(
        new LambdaQueryWrapper<BudgetPreparationDetail>()
            .eq(BudgetPreparationDetail::getSheetId, sheetId)
    );
    
    // 3. 逐条执行校验规则
    for (BudgetValidationRule rule : rules) {
        List<ValidationResult> ruleResults = executeRule(rule, details);
        results.addAll(ruleResults);
    }
    
    return results;
}
```

## 五、审批流程集成

### 1. Flowable流程启动
在预算编制提交时启动流程：

```java
// 启动流程实例
ProcessInstance processInstance = runtimeService.startProcessInstanceByKey(
    "budget_approval_process",  // 流程定义Key
    sheetNo,                     // 业务Key
    variables                    // 流程变量
);

// 更新预算单的流程实例ID
budgetPreparation.setProcessInstanceId(processInstance.getId());
budgetPreparation.setStatus("Pending_Review");
budgetPreparation.setCurrentHandler(nextApprover);
budgetPreparationMapper.updateById(budgetPreparation);
```

### 2. 审批通过任务完成
```java
// 完成任务
taskService.complete(taskId, variables);

// 查询下一个任务
Task nextTask = taskService.createTaskQuery()
    .processInstanceId(processInstanceId)
    .singleResult();

if (nextTask == null) {
    // 流程结束，更新状态为已通过
    budgetPreparation.setStatus("Approved");
} else {
    // 更新当前处理人
    budgetPreparation.setCurrentHandler(nextTask.getAssignee());
}
```

### 3. 审批驳回任务回退
```java
// 驳回时需要回退到上一个节点
runtimeService.createChangeActivityStateBuilder()
    .processInstanceId(processInstanceId)
    .moveActivityIdTo(currentActivityId, previousActivityId)
    .changeState();

// 更新状态和驳回信息
budgetPreparation.setStatus("Rejected");
budgetPreparation.setRejectReason(reason);
budgetPreparation.setRejectLevel(rejectLevel);
budgetPreparation.setCurrentHandler(originalCreator);
```

## 九、注意事项

### ⚠️ 重要提醒
1. **驳回理由必填** - 前端和后端都要校验
2. **数据权限** - 必须在后端Mapper层添加 `@DataPermission` 注解
3. **状态流转** - 确保状态变更符合业务流程
4. **流程集成** - 审批操作需要同步更新Flowable流程状态
5. **通知机制** - 审批结果需要通知编制人

### 🔧 常见问题
1. **部门树加载失败** - 检查 `/api/system/dept/treeselect` 接口
2. **数据权限不生效** - 检查角色数据范围配置
3. **校验结果不显示** - 检查校验规则是否启用
4. **批量审批失败** - 检查是否有记录已被其他人审批
